

```bash
docker compose down
```

## 13.9 이상치 분석

추천 데이터에서 이상치의 의미를 설명하고 분석합니다.

예를 들면 다음과 같습니다.

* 평점을 지나치게 많이 남긴 사용자
* 평점이 지나치게 적은 사용자
* 평가 수가 지나치게 많은 영화
* 평가 수가 지나치게 적은 영화
* 평균 평점은 높지만 평가 수가 매우 적은 영화

Histogram과 Box Plot을 활용합니다.

IQR 등을 무조건 적용하여 데이터를 삭제하지 않습니다.

각 이상치에 대해 다음을 판단합니다.

* 실제 오류인가
* 정상적인 극단값인가
* 삭제할 필요가 있는가
* 필터링 기준으로 사용할 것인가
* 추천 품질에 어떤 영향을 미치는가

## 13.10 왜도 분석

수치형 분포에 대해 왜도를 계산하고 해석합니다.

다음 내용을 포함합니다.

* 왜도의 개념
* 양의 왜도
* 음의 왜도
* 대칭 분포
* 왜도가 큰 컬럼
* 왜도가 추천 시스템에 미치는 영향

왜도가 있다는 이유만으로 모든 값에 로그 변환을 적용하지 않습니다.

사용자 활동량이나 영화 평가 수와 같이 긴 꼬리를 가진 분포에 대해서는 로그 스케일 시각화가 해석에 도움이 되는지 확인합니다.

변환이 필요하다면 적용 이유와 적용 전후 결과를 비교합니다.

# 14. Feature Engineering

추천 시스템에 적합한 Feature Engineering을 설계합니다.

Feature Engineering은 단순히 컬럼을 많이 추가하는 작업이 아니라 추천 품질을 개선하기 위한 데이터 표현 과정으로 작성합니다.

다음 항목을 검토하고, 실제 데이터와 최종 알고리즘에 필요한 항목만 적용합니다.

## 14.1 영화별 평점 개수

예시 컬럼:

```text
rating_count
```

각 영화가 받은 평점 수를 계산합니다.

평균 평점이 같더라도 평점 2개를 받은 영화와 평점 500개를 받은 영화는 신뢰도가 다르다는 점을 설명합니다.

---

## 14.2 영화별 평균 평점

예시 컬럼:

```text
rating_mean
```

인기도 기반 추천과 영화 품질 분석에 활용합니다.

평균 평점만으로 추천할 때 발생하는 문제도 설명합니다.

---

## 14.3 최소 평점 수 기준

평가 수가 너무 적은 영화가 과도하게 높은 순위를 받는 문제를 줄이기 위한 기준을 검토합니다.

기준값을 무조건 임의로 정하지 말고 데이터 분포를 확인하여 합리적으로 결정합니다.

적용 전후 포함되는 영화 수를 비교합니다.

---

## 14.4 사용자 활동량

예시 컬럼:

```text
user_rating_count
```

사용자별 평점 개수를 계산합니다.

활동량이 너무 낮은 사용자가 협업 필터링에 미치는 영향을 설명합니다.

## 14.5 영화 제목 정리

* 검색용 소문자 제목 생성

```text
clean_title
release_year
search_title
```

## 14.6 개봉연도 추출

영화 제목 또는 개봉일에서 개봉연도를 추출합니다.

## 14.9 영화 Feature 행렬

# 18. train_model.py

Notebook에서 최종 선정한 추천 알고리즘을 `train_model.py`에 반영합니다.

Notebook의 실험 코드를 그대로 복사하지 않습니다.

운영에 필요한 코드만 정리하여 작성합니다.

## 필수 처리 과정

```text
데이터 경로 확인
    ↓
MovieLens 데이터 로딩
    ↓
데이터 정제
    ↓
최종 Feature Engineering
    ↓
최종 추천 모델 생성
    ↓
운영에 필요한 산출물 저장
    ↓
모델 정보 저장
```

## 필수 원칙

* Notebook에서 선정한 최종 알고리즘과 동일해야 합니다.
* Notebook과 Feature Engineering 로직이 달라지지 않아야 합니다.
* 고정된 임시 데이터를 사용하지 않습니다.
* 경로를 안전하게 처리합니다.
* 모델 저장 폴더가 없으면 생성합니다.
* 기존 산출물이 있으면 명확한 기준에 따라 덮어씁니다.
* 학습 완료 후 저장된 파일 경로를 출력합니다.
* 데이터 개수와 모델 정보를 출력합니다.
* 예외 발생 시 이해 가능한 오류 메시지를 제공합니다.
* 모든 함수에는 역할을 설명하는 docstring을 작성합니다.

추천 시스템에서 단일 `model.pkl`만으로 충분하지 않다면 필요한 산출물을 나누어 저장합니다.

예:

```text
models/
├── similarity_matrix.pkl
├── movie_metadata.pkl
├── movie_features.pkl
└── model_info.json
```

FastAPI가 모델을 다시 학습하지 않고 저장된 산출물만 로드하여 추천할 수 있어야 합니다.


Pydantic 요청 모델을 사용합니다.


응답 예시:

```json
    "movie_id": 1,

  ]
```

필요하면 다음 API도 제공할 수 있습니다.

```http
GET /recommend/{movie_id}
```

단, 중복 로직을 만들지 말고 내부 추천 함수를 공통으로 사용합니다.

# 32. 강사용 파일 문체

강사가 학생에게 실제로 설명하는 자연스러운 말투로 작성합니다.

다음과 같은 표현을 적절하게 활용합니다.

```text
여기서 학생들이 가장 많이 헷갈리는 부분은 무엇일까요?

이 그래프를 보고 단순히 이상치가 많다고 결론 내리면 안 됩니다.

왜 평균 평점만으로 영화를 추천하면 문제가 생길까요?

실무에서는 이런 데이터를 바로 제거하지 않고 먼저 발생 원인을 확인합니다.

이 Feature를 추가한 이유는 단순히 컬럼 수를 늘리기 위한 것이 아닙니다.

추천 시스템에서는 일반적인 분류 문제와 데이터 해석 방식이 다릅니다.
```

반복적인 감탄이나 불필요한 수식어는 사용하지 않습니다.

강사가 그대로 읽어도 자연스러운 완성된 문장으로 작성합니다.
